# interrupted-session-recovery

## 适用场景
- 当用户执行了 `/new` 或 `/reset` 指令重置会话时。
- 当模型因超时、崩溃或切换而丢失短期上下文内存时。
- 当任务跨越较长时间，需要从之前留下的工作成果中快速定位进度时。

## 步骤
1. **全量扫描工作区**：使用 `ls -R` 或 `find` 命令列出当前目录下所有文件，重点关注带有版本号（如 `v1`, `v2`）或日期后缀的文件。
2. **识别核心进度文件**：寻找名为 `TODO.md`, `PROGRESS.md`, `PLAN.md` 或类似的任务追踪文件。
3. **分析最新状态**：
   - 查看最近修改的文件内容（使用 `ls -lt` 排序）。
   - 读取最后一个版本的文件（如 `script_v3.py` 而非 `script_v1.py`）。
   - 检查是否有未完成的临时文件或构建产物。
4. **重建上下文映射**：根据文件内容推断：
   - 已经完成了哪些步骤？
   - 哪些步骤正在进行中？
   - 下一个逻辑断点在哪里？
5. **主动提议接续**：向用户简要汇报识别到的进度，并直接提出下一步行动建议，而不是询问“我该做什么”。

## 工具和命令
- **查看文件列表及时间戳**：
  `ls -lahtR`
- **查找版本化文件**：
  `find . -maxdepth 2 -name "*_v*"`
- **快速预览进度记录**：
  `cat PROGRESS.md` 或 `tail -n 20 current_task_log.txt`
- **检查 Git 状态（如果适用）**：
  `git status` 和 `git log -n 1`

## 注意事项
- **不要盲目重做**：即使会话是全新的，只要文件系统中有产物，就应视为已完成工作。
- **验证一致性**：检查最新版本的代码是否能正常运行，防止恢复到一个损坏的状态。
- **区分版本**：如果存在多个版本，始终以版本号最高或修改时间最近的文件为准。
- **保持静默分析**：在启动阶段自动执行这些检查，不要在恢复前向用户发送大量询问。

## 示例
**场景**：用户输入 `/new` 后，系统检测到工作区。

**执行过程**：
1. 执行 `ls -F` 发现：`data_clean_v1.py`, `data_clean_v2.py`, `cleaned_data.csv`。
2. 执行 `cat data_clean_v2.py` 发现脚本末尾有 `# TODO: Start visualization`。
3. **回复用户**：
   > 检测到之前已完成数据清洗工作（产物：`cleaned_data.csv`）。根据 `data_clean_v2.py` 的记录，我将现在开始编写可视化分析脚本，您看可以吗？
